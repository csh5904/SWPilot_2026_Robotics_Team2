3-10. 실제 로봇으로 — URDF 완성과 ROS2 추상화

Chapter 3 · 문제 10

과제 목표
로봇 모델에 아두이노 마이크로보드와 모터 드라이버에 해당하는 링크를 추가해 URDF를 완성한다. ROS2의 추상화 수준을 조사하고, 왜 마이크로보드·모터 드라이버를 저수준으로 직접 제어할 필요가 없는지, 시뮬레이션용 제어 프로그램을 실제 로봇에 그대로 쓰는 방법을 알아본다.

1. 마이크로보드·모터 드라이버 링크 추가

제어 보드(OpenCR 등 아두이노 계열 마이크로보드)와 모터 드라이버는 로봇 몸체에 실제로 붙어 있는 부품이다. 이들은 움직이는 관절이 아니므로, 외형·위치·연결 구조만 fixed 조인트로 반영한다. 아래 완성 URDF에서 보드는 받침대 위에 박스 형태로 배치했다(예: 1층 받침대 위 OpenCR 보드 plate1_link 내부 시각 요소).

2. ROS2의 추상화 수준

ROS2는 하드웨어의 세부 동작을 여러 계층으로 감싸서, 위 계층에서는 세부를 몰라도 되게 한다.

계층하는 일예
응용(제어 로직)"어디로 얼마나 갈지" 명령라인트레이싱 노드 → /cmd_vel
메시지·토픽표준 인터페이스로 노드 간 통신geometry_msgs/Twist
드라이버·플러그인명령을 실제 모터 신호로 변환diff_drive, 하드웨어 인터페이스
하드웨어실제 모터·엔코더 구동모터 드라이버, 아두이노

URDF의 모든 링크마다 대응 노드를 반드시 만들 필요는 없다. 움직임·통신이 필요한 부분(바퀴, 센서)만 노드로 다루고, 단순 구조물(받침대, 기둥)은 시각·물리 정보만 두면 된다.

3. 왜 저수준 제어가 필요 없나

제어 노드는 /cmd_vel에 "선속도·각속도"만 보낸다. 이 명령이 실제로 각 모터를 몇 rpm으로 돌릴지, PWM 신호를 어떻게 줄지는 아래 계층(하드웨어 인터페이스, 모터 드라이버 펌웨어)이 알아서 한다.

  • 표준 메시지(Twist, JointState)와 하드웨어 추상화 계층이 세부 신호 처리를 대신한다.
  • 그래서 응용 개발자는 아두이노 핀·타이머·PWM 같은 저수준을 건드리지 않고 "무엇을 할지"에만 집중할 수 있다.
  • 모터나 드라이버가 바뀌어도, 그 부분의 드라이버만 교체하면 위쪽 제어 로직은 그대로 쓸 수 있다.

4. 시뮬레이션 → 실제 로봇 (Sim-to-Real)

Gazebo에서 쓰던 제어 프로그램을 실제 로봇에 거의 그대로 쓸 수 있는 건, 양쪽이 같은 토픽 인터페이스를 쓰기 때문이다.

  • 시뮬레이션: 제어 노드 → /cmd_vel → Gazebo diff_drive 플러그인 → 가상 바퀴
  • 실제 로봇: 제어 노드 → /cmd_vel → 하드웨어 인터페이스(마이크로보드 통신) → 진짜 모터

/cmd_vel을 구독하는 대상만 플러그인에서 실제 하드웨어 드라이버로 바꾸면, 제어 로직(라인트레이싱·PID)은 수정 없이 동작한다. 센서 쪽도 마찬가지로 /scan, /joint_states 같은 표준 토픽을 그대로 받는다.

5. 완성된 URDF (tracked_robot.urdf)

트랙형 이동 로봇의 완성 URDF다. 받침대 3층 구조에 OpenCR 보드·Jetson·배터리·라이다·카메라·IMU를 배치하고, 트랙 중앙에 숨김 구동륜 2개(차동 구동)와 캐스터 구 4개로 물리를 잡았다. Gazebo 플러그인(diff_drive, joint_state, IMU, LiDAR, Depth 카메라)까지 포함한다.
